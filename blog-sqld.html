DATABASE ARCHITECTURE & TUNING

SQLD 및 데이터베이스 심층 학습 노트

안정적인 트랜잭션 처리와 대규모 트래픽을 견디는 쿼리 최적화의 기록

"백엔드 시스템의 병목 현상 90%는 데이터베이스 I/O에서 발생합니다."
단순히 테이블을 만들고 데이터를 꺼내오는 것을 넘어, Spring Boot(Java)와 연동되었을 때 데이터가 어떻게 정합성을 유지(ACID)하고, 어떻게 B-Tree 인덱스를 타며, 실행 계획(Execution Plan)이 어떻게 수립되는지를 원리 단계부터 파헤칩니다.

Phase 1. Architecture

데이터 모델링과 트랜잭션 아키텍처

1. 정규화(Normalization)와 반정규화

데이터베이스 설계 시 '쓰기(Write) 성능'과 '읽기(Read) 성능'의 트레이드오프를 조율합니다.

  • 제1~3 정규화: 중복 데이터를 제거하여 '삽입/갱신/삭제 이상(Anomaly)'을 방지. 데이터 무결성이 최우선인 시스템에 적용.
  • 반정규화(De-normalization): 잦은 JOIN으로 인해 조회(Select) 성능이 심각하게 저하될 때, 시스템 성능 향상을 위해 의도적으로 중복을 허용하는 기법.

2. 트랜잭션(Transaction)과 ACID 특성

백엔드 비즈니스 로직(예: 송금 시스템)에서 절대 깨져서는 안 되는 4가지 원칙입니다.

  • Atomicity (원자성): ALL or NOTHING. 모두 성공하거나 모두 실패해야 함.
  • Consistency (일관성): 트랜잭션 전후로 DB 제약조건이 일관되게 유지되어야 함.
  • Isolation (고립성): 실행 중인 트랜잭션에 다른 트랜잭션이 간섭할 수 없음.
  • Durability (영속성): 완료된 결과는 영구적으로 저장되어야 함.
Phase 2. Query Optimization

다중 테이블 연산과 조인(JOIN) 최적화

1. 카테시안 곱(Cartesian Product)의 위험성

조인(JOIN) 조건을 실수로 누락하거나 잘못 설정하면 CROSS JOIN이 발생하여 두 테이블의 행을 곱한(M * N) 방대한 데이터가 메모리에 올라옵니다. 이는 백엔드 서버 메모리 초과(OOM) 및 DB 락(Lock)의 주원인이 되므로, 항상 연결 고리(ON 절)의 인덱스 여부를 확인해야 합니다.

2. 위치에 따른 서브쿼리(Subquery)의 활용

복잡한 비즈니스 로직을 처리할 때 JOIN만으로 해결하기 어려운 경우 서브쿼리를 전략적으로 배치합니다.

인라인 뷰(In-Line View) & 스칼라 서브쿼리(Scalar Subquery) 예시
SELECT 
    E.EMP_ID, 
    E.EMP_NAME,
    -- SELECT 절 (스칼라 서브쿼리): 단일 행/단일 컬럼 반환. 성능 주의 요망.
    (SELECT D.DEPT_NAME FROM DEPARTMENTS D WHERE D.DEPT_ID = E.DEPT_ID) AS DEPT_NAME,
    V.TOTAL_SALES
FROM EMPLOYEES E
JOIN (
    -- FROM 절 (인라인 뷰): 메모리에 임시 테이블(Dynamic View)을 생성하여 조인
    SELECT EMP_ID, SUM(AMOUNT) AS TOTAL_SALES
    FROM SALES
    GROUP BY EMP_ID
) V ON E.EMP_ID = V.EMP_ID;
Phase 3. Performance Tuning

백엔드 성능의 핵심: 인덱스(Index) 튜닝

1. B-Tree (Balanced Tree) 인덱스 아키텍처

관계형 데이터베이스(MySQL, Oracle 등)에서 가장 널리 쓰이는 인덱스 구조입니다. Root-Branch-Leaf 노드로 구성되며, 데이터를 탐색할 때 O(log N)의 시간 복잡도를 보장합니다. 무분별한 인덱스 생성은 오히려 INSERT/UPDATE/DELETE 시 트리의 재정렬(Index Split) 연산을 유발하여 쓰기 성능을 크게 저하시키므로, 검색 빈도와 카디널리티(분포도)를 고려한 신중한 설계가 필수적입니다.

2. 안티 패턴: 인덱스를 타지 못하는 Full Table Scan

인덱스가 생성된 컬럼이더라도, WHERE 절에서 컬럼 자체를 가공(함수 사용, 형변환)하면 DB 옵티마이저가 인덱스를 포기하고 테이블 전체를 뒤지게 됩니다.

❌ 비효율적인 쿼리 (인덱스 사용 불가)
-- 컬럼(ORDER_DATE)을 SUBSTR로 가공함
SELECT * FROM ORDERS 
WHERE SUBSTR(ORDER_DATE, 1, 4) = '2026';

-- 묵시적 형변환 발생 (문자열 컬럼에 숫자 대입)
SELECT * FROM USERS 
WHERE USER_PHONE = 01012345678;
✅ 성능 최적화 쿼리 (Index Range Scan 유도)
-- 컬럼을 있는 그대로 두고 상수항을 가공
SELECT * FROM ORDERS 
WHERE ORDER_DATE >= '20260101' 
  AND ORDER_DATE < '20270101';

-- 타입 일치 (문자열은 문자열로)
SELECT * FROM USERS 
WHERE USER_PHONE = '01012345678';
